# CLAUDE.md

Guidance for Claude Code (claude.ai/code) when working in this repository.

## What this repo ships

One cross-platform installer that puts a self-hosted web app on a user's
own machine. No hosted control plane, no account, no telemetry. A user
runs `curl | sh` (or the PowerShell equivalent); the installer downloads
the source tarball for the latest GitHub release, builds it under Docker,
and starts it locally. Anyone who wants a public domain in front of it
either picks the domain option in the wizard (auto-Caddy on Linux) or
runs their own reverse proxy.

Two products, released together under the same `vX.Y.Z` tag:

- `apps/web` — the Next.js 14 app end users actually see.
- `installer/` — the Go CLI (`octopus`) that manages the app's lifecycle.

## Commands

### Web app (`apps/web`)

From the repo root. pnpm 9.12.0 (via corepack), Node >=20.11.0.

- `pnpm install`
- `pnpm dev` — localhost:3000
- `pnpm build` — standalone production build
- `pnpm lint` / `pnpm typecheck`

### Installer (`installer/`)

From `installer/`. Stdlib only — no third-party Go deps (`go.sum` stays
empty on purpose; it makes release auditing trivial).

- `go build ./...`
- `go vet ./...`
- `go build -o /tmp/octopus ./cmd/octopus && /tmp/octopus help`
- `go build -ldflags "-X .../version.Current=vX.Y.Z" ...` — goreleaser
  runs the same ldflag so the shipped binary reports its tag.

No test suite yet on either side.

## Architecture

### Monorepo

pnpm workspace (`pnpm-workspace.yaml`):

- `apps/*` — deployable apps (just `apps/web` today).
- `packages/*` — reserved for shared libs; directory doesn't exist yet.
- `installer/` — Go module, intentionally outside the pnpm workspace
  so goreleaser can use `workdir: installer` without Node tooling.

`tsconfig.base.json` at the root sets strict options including
`noUncheckedIndexedAccess` and `noImplicitOverride`. Every package
extends it; `apps/web` adds the `@/*` → `./src/*` alias.

### Web app

Next.js 14 App Router + React 18 + Tailwind. Today a single landing page.

- `src/components/Logo.tsx` is the reusable wordmark component. Its size
  is `100vmin / 3.8125` — derived from the octopus-pixel glyph widths +
  an 8-font-pixel padding on each side — so the logo's bounding box
  hugs the short edge of any viewport. `vmin` makes that immune to
  orientation changes: rotate a phone portrait↔landscape and the
  apparent size stays the same. An explicit `size` prop overrides.
- `src/app/layout.tsx` loads `octopus-pixel.ttf` via `next/font/local`,
  exposes it as `--font-octopus-pixel` → Tailwind `font-pixel`.
  `globals.css` disables font smoothing on `.font-pixel` and locks the
  body to `100dvh` so there's no fake scroll on mobile browsers.
- `octopus-pixel.ttf` is a derivative of basis33 (MIT) generated by a
  one-shot fontTools script. Letters, punctuation, and Cyrillic are
  proportional with exactly **2 visual pixels** (128 font-units on
  basis33's 1024-upm grid) between their bounding boxes; digits stay
  monospace at 7 pixels (448 units); space is 5 pixels. The font's
  `hhea.descent` / `OS/2.usWinDescent` are trimmed to -192/192 units
  so the em-box bottom coincides with the lowest descender pixel —
  needed by `page.tsx` to line the version badge's baseline up with
  the bottom of "p".
- `next.config.mjs` reads two env vars at build time:
  - `OCTOPUS_BASE_PATH` → Next's `basePath`. Installer passes it as a
    docker build arg so the user's chosen subpath is baked into the
    standalone bundle.
  - `OCTOPUS_VERSION` → exposed to the client as
    `NEXT_PUBLIC_OCTOPUS_VERSION`; rendered under the logo.
- `next.config.mjs` also sets `output: "standalone"` and points
  `experimental.outputFileTracingRoot` at the monorepo root so the
  standalone bundle picks up the correct slice of root `node_modules`.
  **Do not remove** — the Dockerfile depends on the path
  `apps/web/server.js`.

### Installer

Go module `github.com/mk-amorson/Octopus/installer`.

```
cmd/octopus/main.go              switch over os.Args[1]; no cobra/flag magic.
internal/version/version.go      installer version; overridden at release via -ldflags -X.
internal/state/state.go          ~/.octopus layout, config.json schema.
internal/docker/docker.go        Run/RunQuiet/Compose/ComposeQuiet wrappers.
internal/docker/install.go       Ensure(): detect/install Docker per OS.
internal/source/source.go        Download release source tarball; LatestTag().
internal/selfupdate/selfupdate.go SHA256-verified CLI self-replace + re-exec.
internal/wizard/wizard.go        stdin prompts for domain / host / subpath / port.
internal/stack/stack.go          Render docker-compose.octopus.yml; build/up/down.
internal/caddy/caddy.go          Install Caddy + reverse-proxy a domain (Linux).
internal/commands/install.go     install flow: docker → wizard → source → build → up → Caddy.
internal/commands/lifecycle.go   start, stop, status.
internal/commands/update.go      Self-update CLI if behind; then rebuild app.
internal/commands/uninstall.go   down -v + image rm + Caddyfile cleanup + state wipe.
```

Design choices worth knowing:

- **One install per user**, keyed off `~/.octopus` (on Windows,
  `%USERPROFILE%\.octopus` via `os.UserHomeDir`).
- **Source-tarball flow.** The installer pulls
  `https://codeload.github.com/<repo>/tar.gz/refs/tags/vX.Y.Z` and
  builds the Docker image locally, so the user's chosen subpath can be
  baked into the build without shipping a matrix of pre-built images.
- **`octopus update` is the single upgrade command.** It checks GitHub
  for a newer release; if the CLI itself is out of date, `selfupdate`
  downloads the new binary, verifies its SHA256 against `checksums.txt`,
  atomically rename-replaces the running executable, and `syscall.Exec`s
  into it — then the new CLI rebuilds the app. Self-update stages files
  next to the running exe so `os.Rename` never crosses filesystems.
- **Docker compose project = `octopus`**, container = `octopus-web`
  (`state.ComposeProject` / `state.ContainerName`). Every compose call
  pins `-p octopus`, so one `octopus uninstall` wipes everything cleanly.
- **The rendered compose file is `docker-compose.octopus.yml`** inside
  the extracted source tree. It's overwritten on every install/update;
  never hand-edit.
- **Domain → Caddy.** When the wizard gets a domain, the installer binds
  the container to 127.0.0.1, installs Caddy via apt if missing, writes a
  Caddyfile with `reverse_proxy 127.0.0.1:<port>`, restarts the service.
  Let's Encrypt handles TLS on first request. Linux-only; Mac/Windows
  just skip the domain question.

### Release pipeline

Three workflows:

1. `.github/workflows/tag.yml` — mobile-friendly "Run workflow" button
   that creates and pushes a git tag on `main`, then dispatches
   `release.yml` against it with the tag as an input.
2. `.github/workflows/release.yml` — goreleaser builds six binaries
   (linux/darwin/windows × amd64/arm64) + `checksums.txt` and attaches
   them to a GitHub Release. Accepts a `tag` input (normalised to add
   `v` prefix) and validates the tag exists before checking out.
3. `.github/workflows/pages.yml` — publishes `docs/pages/` to GitHub
   Pages on every push to `main` that touches that directory. The
   site serves the two bootstrap shims (`install`, `install.ps1`) and
   a landing `index.html`.

Bootstrap shims resolve the latest release tag via the `/releases/latest`
HTML redirect (no API rate limits), download + SHA256-verify the matching
binary, drop it in `~/.octopus/bin` (or `%LOCALAPPDATA%\Octopus\bin`),
add to PATH, `exec` `octopus install` with stdin wired to `/dev/tty` so
the interactive wizard actually gets answers.
